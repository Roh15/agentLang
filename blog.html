Latent Agent Communication:
Eliminating Redundant Prefill in Multi-Agent LLM Pipelines

The Problem

Multi-agent LLM pipelines have a hidden tax. When Agent A finishes and passes its output to Agent B, the standard approach decodes Agent A's generation to text—then Agent B re-tokenizes and re-prefills that entire sequence before it can generate its first token. The computation is identical to what Agent A already did. We're paying for it twice.

For a 2-hop pipeline, this means 1 wasted prefill. For a 4-hop pipeline: 3. At production scale, this is not a rounding error.

The fix is conceptually simple: pass the KV cache directly. Agent B's system prompt gets prefilled (small cost, ~41 tokens), then generation continues from the concatenated cache. No re-prefill of Agent A's output. The primitive required is a session_id field on inference calls—enough for a provider to route chained calls to the same process and hand off the cache.

The Mechanism

The KV cache stores the pre-computed key and value tensors for every token in the sequence. In a transformer, the attention computation for each new token requires K and V from all previous tokens—but once computed, they don't change. This is what use_cache=True preserves across generation steps.

The challenge is that K tensors have Rotary Position Embeddings (RoPE) baked into them at the positions where they were originally computed. If Agent A generated tokens at positions 42–550, those K tensors encode absolute positional information from Agent A's context space. Transplanting them directly into Agent B's sequence—where the same tokens should sit at positions 42–550 relative to Agent B's system prompt—requires correcting the positional encoding.

RoPE Re-indexing

The inverse rotation formula undoes the old positional encoding and re-applies the correct one:

# Undo old positions, re-apply correct positions
k_raw  = k_rotated * cos_old - rotate_half(k_rotated) * sin_old
k_new  = k_raw     * cos_new + rotate_half(k_raw)     * sin_new

V tensors are unchanged (RoPE is not applied to values). Only K tensors need re-indexing.

This step is not optional. EXP-002 shows that skipping re-indexing produces completely broken output—not slightly degraded output. The model immediately generates truncated, syntactically invalid code. See the ablation section below.

The floor effect (below ~200 tokens): Standard prefill costs ~92–96 ms regardless of input length—the minimum cost of a forward pass dominated by kernel launch overhead, not sequence length. No saving exists here; both pipelines hit the same floor.

The O(n²) regime (above ~1,000 tokens): The power-law exponent climbs from 0.59 (EXP-003, 50–800 tokens) to 1.38 (EXP-007, 800–4,000 tokens) as quadratic attention cost starts dominating over linear projection cost. The mechanism is most valuable in exactly this regime—long tool outputs, multi-step reasoning chains, large document analysis—where agentic pipelines actually operate.

† The 500-token 3B point (200.4 ms) and 1,000-token 7B point (845.7 ms) each read slightly lower than the preceding measurement. Both are within run-to-run variance (n=10) and likely reflect cache or scheduler artifacts. The dips are not systematic features.

The exponent (~0.59 overall, ~1.28 in the 400–800 range) is lower than the theoretical O(n²) because at <2000 tokens, linear projection cost dominates over quadratic attention cost. At longer contexts—common in real agentic tasks with large tool outputs—the exponent will increase toward 2.0 and savings will compound faster.

Practical operating rule

Agent A must produce >~250 tokens for KV cache passing to deliver meaningful savings. Above that threshold, speedup grows from 1.6× to 4.4× over the 300–800 token range measured here, and continues increasing at longer contexts.

Why does speedup grow with model size? Standard prefill scaled 2.27× (301 ms → 682 ms) from 3B to 7B—it grows with both model depth and sequence length. KV prefix prefill barely changed (93.5 ms → 101.3 ms, +8%)—41 tokens is so short that adding more layers barely affects it; this regime is dominated by fixed kernel overhead. The numerator grows much faster than the denominator.

This pattern extrapolates: standard prefill cost scales with both model depth and sequence length, while KV prefix cost is bounded by fixed kernel overhead at short sequences and grows slowly. Two measured points aren't enough to fit a curve, but the direction is clear—the numerator grows faster than the denominator as models scale.

Actual prefill scaling depends on attention implementation, batching, and hardware memory bandwidth—all of which differ significantly at datacenter scale. The provider argument crystallized here

The gap between 1.4 ms (in-process) and 73 ms (naive serialization) quantifies why providers need a first-class primitive. Ad-hoc serialization works but wastes most of the gain. A session_id API that routes chained calls to the same GPU process eliminates that 73 ms—reducing transfer cost to microseconds via CUDA IPC.

EXP-006 System Prompt Sensitivity — When Does the Mechanism Pay Off?

Speedup is driven by the ratio of what Agent A produced to what Agent B must prefill from scratch. As Agent B's system prompt grows, the KV prefix prefill cost grows and erodes savings. We swept a 4×4 grid of prior context lengths × system prompt lengths (10 runs per cell) on the 7B model.

Figure 5. Speedup heatmap: prior context length (rows) × system prompt length (columns). 15 of 16 cells deliver ≥ 1.6× speedup. Only the 200-token prior / 400-token system prompt cell falls to 1.3×.

The 400-token system prompt column is the warning. Long, detailed system prompts (common in enterprise agentic applications) see the smallest gains—1.3× to 2.2×. This is still a saving, but a smaller one. Pipeline designers should characterize their workload's output_tokens / system_prompt_tokens ratio before committing to the mechanism.

The 100-token column being occasionally higher than the 41-token column (e.g., 200 prior: 3.2× vs 2.4×) is not an error—it's the fixed-overhead floor effect. At short sequences, both standard and KV prefix are in the kernel-overhead regime where adding tokens doesn't cost proportionally yet.

Practical operating rule

Build this when: prior_context_tokens / system_prompt_tokens > 3. Below that ratio, speedup falls under 2× and the engineering overhead may not justify it. Most real agent pipelines (analysis → repair, research → synthesis) naturally exceed this ratio—Agent A writes hundreds of tokens; Agent B has a system prompt measured in tens of tokens.

What Providers Need to Build

The primitive is minimal: a session_id on inference API calls. When Agent B's call carries the same session_id as Agent A's completed call, the provider:

  1. Routes the request to the same GPU process (or uses CUDA IPC to transfer the cache)
  2. Extracts the KV slice corresponding to Agent A's generation tokens
  3. Re-indexes K tensors to correct positional encodings
  4. Prefills only Agent B's system prompt (~41 tokens)
  5. Generates from the concatenated cache

The server-side complexity is real—primarily the RoPE re-indexing step. But the payoff scales with every hop added, every token Agent A generates, and every basis point of GPU cost at production scale.

At production scale on 7B+ models: A 4-hop pipeline saves ~1.7 seconds of prefill per call (3 hops × ~580 ms saving/hop, Jetson timing). At 10M calls/day, that is approximately 4,850 GPU-hours/day of avoidable compute—assuming Jetson-scale timing. Production datacenter hardware prefills faster in absolute terms, so the GPU-hour figure scales accordingly; the speedup ratio (6.74×) is the hardware-agnostic result. The larger the model and the longer the agent outputs, the stronger the case.

Open Questions

  • What is the actual CUDA IPC transfer cost between processes on the same GPU? (EXP-005 used serialization—real providers would use IPC)
  • Does the mechanism generalize to cross-model handoffs with shared tokenizer but different layer counts?
